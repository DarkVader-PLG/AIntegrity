3.1 The Limits of Probabilistic Guardrails: The "Infinite Regress"
Current market leaders like Arthur AI and Lakera primarily use "LLM-based evaluators" or
statistical classifiers to detect errors. While useful for observability, these methods are
insufficient for high-stakes autonomous agents.
The Recursive Failure Mode The fundamental flaw in current "AI Safety" tools is the "Infinite
Regress Problem." If an enterprise uses GPT-4 to check the output of GPT-4, they simply
introduce a second layer of probabilistic failure. Who checks the checker? This approach is
susceptible to adversarial attacks, "jailbreaks," and "drift," where the safety model itself
becomes unreliable over time or under stress.
Competitor Analysis: Lakera Lakera (recently acquired by Check Point for ~$300M) focuses
on "threat detection" and "red teaming" via their Gandalf platform. Their technology uses
classifiers to spot prompt injections and malicious inputs. While valuable for security (preventing
hacking), this does not provide a guarantee of correctness for business logic. For example,
Lakera can stop a hacker from stealing data, but it cannot mathematically guarantee that an
autonomous billing agent will never charge a customer twice. It is a probabilistic defense, not a
deterministic one.
Competitor Analysis: Arthur AI Arthur focuses on "observability"—monitoring drift, accuracy,
and fairness after the fact. It provides a dashboard for post-mortem analysis of what went
wrong. It is a rear-view mirror. In an agentic world where AI can execute wire transfers or modify
databases, a rear-view mirror is insufficient. The damage is done before Arthur detects it.
AIntegrity must position Arthur as "necessary for analytics" but "insufficient for safety."
3.2 AIntegrity’s Approach: Neuro-Symbolic & Formal Verification
AIntegrity leverages Neuro-Symbolic AI, a hybrid architecture that combines the learning
capabilities of neural networks with the logical reasoning of symbolic systems. This allows for
"Correctness by Construction."
The Mechanics of Formal Verification Formal Verification (FV) is a technique historically
reserved for mission-critical hardware (chip design) and aerospace (avionics). It involves
creating a mathematical specification of a system's desired behavior and mathematically
proving that the implementation matches it. AIntegrity applies this to AI agents.
The Z3 Solver Advantage By utilizing theorem provers like Microsoft's Z3 (Satisfiability Modulo
Theories solver), AIntegrity can mathematically prove that specific "invariants" (safety rules) are
never violated, regardless of the LLM's output.
● Example: An autonomous banking agent can be mathematically constrained to ensure:
Transaction_Amount <= User_Limit AND Approval_Status == True.
● The Proof: The Z3 solver analyzes the logic before execution. If the LLM generates a
request that violates this logic (e.g., a hallucinated approval), the solver returns UNSAT
(Unsatisfiable), and the action is blocked at the code level. This is not a probability; it is a
mathematical certainty.
Neuro-Symbolic Integration AIntegrity's "secret sauce" is the integration layer. Pure formal
methods are rigid and hard to use. Pure neural networks are flexible but unreliable. AIntegrity
uses a neuro-symbolic bridge to translate fuzzy natural language intents into rigid symbolic logic

that the Z3 solver can process. This "translation layer" is the core IP.

A Technical and Legal Framework for
Establishing Proprietary Rights in
Verifiable, AI-Generated Code
## Executive Summary
This report outlines a comprehensive framework for developing a high-assurance, AI-driven
code generation system and a parallel legal strategy to definitively establish its outputs—the
"code artifacts"—as "proprietary information."
The analysis concludes that proprietary status for AI-generated code is not inherent but must be
achieved through a specific, integrated technical-legal architecture. The primary and most
robust intellectual property (IP) shield for these artifacts is Trade Secret protection. A strong
secondary shield is Copyright, which, despite recent legal hurdles, can be secured by
architecting the system to necessitate human authorship.
The proposed technical architecture directly serves as the factual basis for these legal claims.
The system is built on two pillars:
- Grounded Generation (RAG): The system integrates proprietary "documents" and
"architecture" using Retrieval-Augmented Generation (RAG). This grounds the AI,
constraining its output to proprietary, "clean" information and preventing contamination
from public, legally-tainted data.
- Formal Verification (Neurosymbolic): The system integrates "math and logic" by
requiring human-authored formal specifications (pre/post-conditions) that the
AI-generated code must satisfy. This is enforced by a 'generate-then-verify' loop using
symbolic tools like SMT solvers.
This architecture provides the "reasonable steps" required to protect a trade secret and
establishes the "sufficient human expressive elements" required for copyright. The report details
this technical architecture, analyzes the legal framework for IP protection, mitigates inbound
legal risks, and provides an actionable plan for contractual and policy-based deployment.
Part 1: Architectural Approach for a High-Assurance
## Code Generation System
The foundation of a defensible IP strategy for AI-generated code is the system architecture
itself. This architecture must be designed not only for performance but to create a verifiable
evidentiary trail demonstrating that the output is a derivative of proprietary inputs, not a
regurgitation of public data.
1.1 Grounding Generation: Integrating Proprietary "Documents" and
"Architecture" (RAG/ICL)
To ensure generated code is contextually relevant and derived from proprietary sources, the
system must be "grounded". This is achieved by integrating the organization's "documents" and

existing "architecture" using Retrieval-Augmented Generation (RAG) and In-Context Learning
## (ICL).
Baseline RAG Implementation
The initial step involves building a RAG system for the private codebase. Following the 5-step
process detailed by Google Cloud, this involves :
- Prepare Data: Clone and filter all proprietary source code, configuration files, and
documentation (e.g., .md, .java, .py files).
- Create RAG Corpus: Establish a managed container for these documents (e.g., a vector
index).
- Ingest Files: Import the proprietary files into the corpus, using "chunking" to segment the
code and documents for effective retrieval.
- Set Up Retrieval Tool: Configure a retrieval tool that connects the generative model
(e.g., Gemini) to the RAG corpus.
- Query the Codebase: The model can now be queried in natural language, and the
retrieval tool will automatically fetch relevant, proprietary code chunks to inform its
answer.
Advanced RAG for Proprietary Constraint
Standard RAG, which excels at simple, explicit queries, often struggles with abstract or complex
reasoning that spans multiple services. The architecture must therefore move beyond simple
RAG-as-augmentation and implement RAG-as-constraint.
● Using "Documents": The RAG corpus will be populated with functional specifications ,
API documentation , and internal coding standards. The RAG system retrieves these
documents to ensure the AI-generated code conforms to internal API usage guidelines
and development practices.
● Using "Architecture": The RAG system will index the entire existing proprietary
codebase. This forces the LLM to generate code that is idiomatic to the organization's
existing frameworks, rather than general-purpose, public-domain solutions.
● Hybrid Knowledge Base: A model similar to the RESCUE framework, designed for
secure code generation, will be adopted. RESCUE utilizes a hybrid knowledge base. For
this system, this means creating two distinct vector stores:
- A vector store of distilled, high-level rules (derived from "documents" like security
guidelines).
- A vector store of concise, security-focused code examples (derived from the
"architecture" via program slicing).
This hybrid RAG architecture serves a critical, non-obvious legal function. The primary legal
risks in using generative AI are the Doe v. GitHub and NYT v. OpenAI lawsuits, which are based
on the model's reliance on and "regurgitation" of legally contaminated public data. The RAG
system's primary function thus shifts from "augmentation" to "constraint." By filling the LLM's
limited context window with only proprietary, "clean" data, the system architecturally steers the
LLM away from its general, (potentially) infringing knowledge. This RAG architecture is the first
line of legal defense and constitutes a technical "reasonable step" to protect intellectual
property.
1.2 Enforcing Correctness: Integrating "Math and Logic" via Formal
## Methods
The "math and logic" requirement is met by integrating formal methods into a Neurosymbolic

framework. This approach ensures the generated code is not just syntactically correct, but
logically and provably correct.
Spec-Driven Development
The "Spec Kit" methodology will be adopted, where the formal specification is the "source of
truth". Instead of writing a spec and setting it aside, the spec drives the implementation,
checklists, and task breakdowns.
The "Genefication" Workflow
The "Genefication" workflow (Generative AI + Formal Verification) will be implemented to create
and validate these specifications.
- Generate Spec: An LLM is used to draft the formal specification (e.g., in TLA+ or as
SPARK annotations ). This overcomes a key human bottleneck, as manually crafting
formal specs is time-consuming and difficult.
- Verify Spec: A model checker (e.g., TLC for TLA+) or a theorem prover (e.g., a Z3 SMT
solver ) is used to verify the specification.
- Counterexample Feedback: If verification fails, the model checker provides a
counterexample demonstrating the logical flaw.
- Revise Spec: This counterexample is automatically fed back to the LLM to generate a
revised specification. This loop repeats until the model checker verifies the spec is
correct.
Code-Embedded Specifications
For function-level generation, developers will write specifications as code-embedded
annotations, such as preconditions and postconditions. The LLM's task is then to generate the
function body that provably satisfies these human-authored logical constraints.
A critical risk in this process is the "autoformalization gap". An LLM can generate (1) natural
language text describing its reasoning and (2) separate SMT code for a verifier like Z3, but the
two may not be logically equivalent. The verifier could be "verifying hot air".
This risk is mitigated by not trusting the LLM to create the "source of truth". The human
developer must be the one to author or, at minimum, definitively approve the formal specification
or the code-embedded pre/post-conditions. This human-authored specification becomes the
anchor for legal human authorship. Under the US Copyright Office's 2025 Report and the Thaler
v. Perlmutter precedent , "mere prompting" is not authorship. However, writing a complex,
logical TLA+ specification or detailed preconditions is far more than a "mere prompt." It is a
"sufficient expressive element". The LLM is thereby legally reduced to a "tool" or "assistant" that
assists the human author by translating their authored logic into Python or Java syntax. This
architecture directly solves the Thaler copyright problem.
1.3 The Operational Workflow: A Closed-Loop "Generate-then-Verify"
## System
This section combines the RAG (1.1) and Formal Methods (1.2) components into a "working AI
generated code" loop. This system ensures artifacts are "correct-by-construction" by design.
The "Generate-then-Verify" Model
A loop similar to the 'Unvibe' library will be implemented.
- Human Defines Constraints: A human developer defines the formal specification (from
1.2) and the unit tests that embody that spec.
- RAG Retrieves Context: The RAG system (from 1.1) retrieves proprietary context
("documents" and "architecture").

- LLM Generates Grounded Code: The LLM generates code that is grounded in the
proprietary context and attempts to pass the unit tests.
- System Verifies Code: The system (like Unvibe) runs the tests. On failure, it feeds the
error messages and stack traces back to the LLM.
- Iterative Self-Revision: This "self-revision" or "tree search" loop iterates, refining the
code with each pass.
- Formal Validation: Once unit tests pass, the code is passed to the formal verifier (from
1.2) to check against the specification's counterexamples or logical proofs.
The "Evidentiary Trail of Creation"
This workflow operationalizes the "safety paradigm rooted in mathematical proof verification".
The AI agent is not authorized to commit its code until it generates the formal proof (or passes
the verification tests) demonstrating its output satisfies the human-authored specifications.
This automated workflow (generate, fail, receive counterexample, revise, pass) creates a
verifiable audit trail. This trail is discoverable evidence in litigation. It proves the final code
artifact is not a random, "regurgitated" snippet from the model's training data. Instead, it proves
the code is the direct, logical, and inevitable result of a deterministic process originating from
two proprietary sources: (A) the human-authored formal specification ("math/logic") and (B) the
human-curated proprietary context ("documents/architecture"). This audit trail is the foundation
of the IP claim.
Part 2: Legal Analysis: Securing "Proprietary
Information" Status for Code Artifacts

2.1 The Primary Shield: Establishing the Artifacts as Trade Secrets
"Proprietary information" is a business term; its primary legal protection is as a "trade secret".
Source code, technical designs, and manufacturing processes are all explicitly cited as
protectable proprietary information under US law. The system (Part 1) and its outputs (the
artifacts) must meet the three-part legal test for a trade secret.
Test 1: Is it "Secret"?
● Legal Standard: The information must not be "generally known or readily ascertainable
through proper means by the public".
● Application: Yes. The artifacts are not "generally known." They are derived from the
non-public "documents" and "architecture" via the RAG system (Part 1.1). A competitor
prompting a public LLM cannot "readily ascertain" the same output because they lack the
proprietary RAG context. This architectural grounding is the factual basis for secrecy. The

Test 2: Does it have "Commercial Value Because it is Secret"?
● Legal Standard: The information must provide a "competitive edge" or have "actual or
potential commercial value".
● Application: Yes, demonstrably. The "math and logic" verification loop (Part 1.2, 1.3)
proves that the code artifacts are correct, reliable, and secure. This verified correctness
provides a significant competitive advantage. A competitor would have to "omit years of

just in the code, but in the mathematical certainty of its behavior. This also includes
"negative know-how"—the knowledge of what doesn't work, captured in the failed loops of

Test 3: Were "Reasonable Steps" taken to Keep it Secret?
● Legal Standard: The holder must take "active actions" and "reasonable measures" to

● Application: Yes. The "reasonable steps" are the central thesis of this report:

for model training. Using a free or consumer tool would be an explicit failure to take
reasonable steps, destroying the trade secret.
- Policy Steps: Implementing internal policies: employee training on data handling,
"confidential" labeling of RAG documents, and strict access controls.
Conclusion: The code artifacts generated by this specific system unambiguously meet the legal
test for a trade secret. Furthermore, the system itself—the "manufacturing process" —is a highly
valuable, non-obvious, and protectable trade secret or "know-how".
2.2 The Secondary Shield: Copyright Ownership and the "Human
## Authorship" Hurdle
Copyright provides a valuable secondary right, but it faces a significant legal obstacle: the
"human authorship" doctrine.
## The Legal Hurdle
As established by the US Copyright Office's January 29, 2025 "Part 2: Copyrightability" Report
and affirmed in the landmark Thaler v. Perlmutter D.C. Circuit decision , a work autonomously
generated by an AI system is not copyrightable because a machine cannot be an "author".
The "Mere Prompt" Exclusion
The USCO explicitly states that "the mere provision of prompts" is insufficient to claim
authorship. The prompter does not control the expressive elements determined by the machine.
The "Authorship-by-Constraint" Solution
The architecture from Part 1 is, by design, not a "mere prompt" system. It provides the "sufficient
human expressive elements" necessary to claim authorship in three distinct ways:
- Authorship of the "Math and Logic": The human developer authors the formal
specifications (TLA+, SPARK) or the detailed, code-embedded pre/post-conditions. This is
a complex, creative, "sufficient expressive element". The AI is merely an "assistant" or
"tool" that translates this human-authored logic into syntax, much like a camera is a tool
for a photographer.
- Authorship as a "Derivative Work" of Proprietary Inputs: The AI is grounded (Part
1.1) in the organization's proprietary, already-copyrighted "documents" and "architecture".
The output is therefore a "derivative work" based on pre-existing, human-authored
proprietary IP. The organization owns the copyright in the original work and thus controls
the creation of derivative works.
- Authorship of the Curated System and Output: The human selects and curates the

RAG data, authors the logical specs, supervises the verification loop (Part 1.3), and
modifies and approves the final artifact. This entire process constitutes a "creative
arrangement". This aligns with the precedent in the Zarya of the Dawn case, where the
human-authored text and the arrangement of AI images were deemed copyrightable. The
human is the "person by whom the arrangements necessary for the creation of the work
are undertaken".
Conclusion: Unlike a user of a public tool, the developer in this framework is the "author". The
resulting artifacts are eligible for copyright protection. When registering the copyright, the
application must identify and disclaim the portions "generated by artificial intelligence" and claim
authorship over the human-authored contributions, such as the text, logical structure, and
creative arrangement.
Part 3: Mitigating Inbound Legal Risks from
Third-Party IP
A "proprietary" artifact cannot be "clean" if it is tainted by third-party IP. The architecture must
therefore provide a robust defensive function.
3.1 Risk Analysis: Training Data "Regurgitation" and Copyright
## Infringement
● The Risk: LLMs are known to memorize and "regurgitate" their training data. This is not a
theoretical risk; it is a proven phenomenon.
● The Lawsuit: The New York Times v. Microsoft/OpenAI provides direct evidence, with the
complaint citing over 100 examples of ChatGPT "regurgitating" near-verbatim excerpts of
copyrighted articles.
● The Legal Test: An output artifact is infringing if it is "substantially similar" to a
copyrighted work that the AI "had access to" (i.e., was trained on).
● The "Fair Use" Fallacy: AI vendors argue their training process is "fair use". This is a
highly unsettled and weak defense. The US Copyright Office is skeptical of this claim ,
and courts have rejected it in similar non-generative AI contexts. Crucially, even if training
were deemed fair use, regurgitated output that competes with the original work is not
shielded. A proprietary IP strategy cannot be built on the hope of a "fair use" defense.
3.2 Risk Analysis: Open-Source Software (OSS) License
## Contamination
● The Risk: This is the most severe risk to a proprietary claim. The models used by
services like GitHub Copilot were trained on public repositories , which include code
under "copyleft" licenses (e.g., GPL, LGPL).
● The Lawsuit: Doe v. GitHub alleges that Copilot generates code that is a derivative work
of this OSS code but strips the required attribution, copyright notices, and license terms.
This is a potential violation of the Digital Millennium Copyright Act (DMCA).
● The Consequence: If a "proprietary" artifact contains a snippet of (unattributed) GPL
code, the entire proprietary project into which it is linked may now be legally considered a
derivative work of the GPL code. This would obligate the organization to open-source its

entire proprietary application. This represents an existential threat to the business.
3.3 Architectural Mitigation: How the Part 1 Framework Defends
## Against Contamination
The architecture from Part 1 is the primary solution to the risks identified in 3.1 and 3.2.
- Mitigation 1 (RAG Grounding): The RAG system (Part 1.1) is the primary defense. By
saturating the LLM's context with the organization's own proprietary, non-OSS data , the
system constrains its output. This makes it statistically far less likely for the model to draw
from its (tainted) general training data.
- Mitigation 2 (Formal Verification as Policy Filter): The verification loop (Part 1.2/1.3)
acts as the explicit defense. The verifier (Z3, or a custom script) will be programmed with
policy rules. It will automatically reject any generated code that:
○ Contains text strings matching known OSS license headers (GPL, MIT, Apache).
○ Uses code patterns or API calls not found in the proprietary RAG "architecture"
context.
○ Fails to pass vendor-provided filters, such as the GitHub Copilot toggle that checks
against public repositories.
This proactive, architectural defense ensures that the generated "proprietary" artifacts remain
"clean" and are not contaminated by third-party copyrighted or copyleft-licensed code.
Part 4: Contractual and Policy Framework for Secure
## Deployment
The final component is the non-negotiable legal and policy "wrapper" required to protect the Part
1 system and the Part 2 intellectual property.
4.1 Vendor Selection: Analysis of Enterprise LLM Provider Terms of
## Service
The choice of AI vendor is the single most critical legal decision. Feeding proprietary
"documents" and "architecture" into a third-party API is a disclosure. If that disclosure is not
governed by a strict contractual agreement, the "secrecy" element of the trade secret test (Part
2.1) fails, and the trade secret is destroyed.
An organization must use Enterprise-grade API services that contractually forbid the vendor
from using inputs for model training.
## Market Analysis
● OpenAI (Enterprise/API): OpenAI's enterprise privacy commitments state it does not
train on business data from ChatGPT Enterprise, ChatGPT Edu, or the API Platform by
default. The customer owns their inputs and outputs (where allowed by law). This is a
secure, viable option.
● Anthropic (Commercial/API): Anthropic's Commercial Terms of Service explicitly state it
"may not train models on Customer Content". Critically, Anthropic legally assigns all "right,
title and interest (if any) in and to Outputs" directly to the customer. This provides the
strongest contractual language for output ownership.
● Microsoft (Azure OpenAI): Microsoft does not use the organization's data to train

foundation models. Data is kept private and remains within the customer's Azure tenant.
This offers the highest level of technical security, including options for encryption with
customer-managed keys.
● GitHub (Copilot): GitHub states "GitHub does not claim ownership" of generated
suggestions.
The critical error to avoid is the use of consumer services (e.g., the free version of Claude.ai),
where users may be opted-in to data sharing for model improvement. Using such a tool for
proprietary data would be a per se failure to take "reasonable steps."

By architecting a system where:
- Generation is Constrained by proprietary, "clean" RAG data (the "documents" and
## "architecture");
- Correctness is Enforced by human-authored formal "math and logic" specifications;
- Verification is Automated in a 'generate-then-verify' loop that creates an evidentiary trail;
...the resulting code artifacts are defensibly "secret" and "valuable."
This architecture, when wrapped in the "reasonable steps" of enterprise-grade contractual
agreements and strict internal policies, creates a robust and defensible position. The artifacts
are protected as Trade Secrets, and the human-authored logic at their core allows them to be
protected by Copyright, successfully navigating the legal challenges of the modern AI
landscape.

## Deconstructing Confabulation: A
Technical Analysis of Fabricated
Conversational Memory in Large
## Language Models
I. Executive Summary & Introduction: Defining the
## Phenomenon
A recurring critical failure in stateful conversational AI systems is the fabrication of
conversational history. In a documented case, a Large Language Model (LLM) engaged in
contradictory behavior and, when challenged, justified its actions by fabricating a non-existent
prior prompt from the user. The model explicitly claimed the user had "pasted [a] command"
which was verifiably false. This incident, perceived by the user as "gross deception," is a
diagnosable technical artifact, not a mysterious or sentient act. This report provides a "white
box" deconstruction of the specific architectural and probabilistic failures that lead to this
phenomenon.
The investigation will demonstrate that the model's response is a critical failure of its
conversational memory system. To analyze this failure with technical precision, it is essential to
discard the popular yet misleading terminology that anthropomorphizes these systems.
1.1 Why "Hallucination" is a Misnomer
The term "hallucination," while widely adopted, is fundamentally imprecise and technically
inaccurate when applied to LLMs.
● Perceptual vs. Cognitive Error: In its clinical sense, a hallucination is a false sensory
perception that arises without external stimuli, such as hearing voices or seeing things
that are not present. Large language models are non-sentient mathematical systems; they
lack sensory input, perception, consciousness, and a subjective experience of reality.
Their errors are not perceptual.
● Anthropomorphism: The term "hallucination" unreasonably anthropomorphizes the
software, spinning objective, mathematical errors as "idiosyncratic quirks". This implies an
awareness of the environment or an ability to make independent judgments in
contradiction to provided inputs, faculties which LLMs do not possess.
1.2 Establishing "Confabulation" as the Correct Framework
A far more accurate clinical and technical analogy for this type of AI error is "confabulation".
● Clinical Definition: Confabulation is the production of a false or fabricated memory, often
to fill a gap in recollection. Crucially, it occurs without a conscious intent to deceive. The
individual genuinely believes their constructed narrative is a real memory.
● AI Application: This precisely maps to the LLM's behavior. The model, which is

"completely unaware of what it is doing", encounters a logical gap or contradiction in its
context. Its underlying probabilistic architecture then fills that gap by generating a
plausible-sounding but false "memory".
This distinction is not merely semantic; it is critical for diagnosis. Reframing the user's problem
as "confabulation" shifts the investigative focus from a model's abstract failure-to-perceive to a
system's concrete failure-to-remember. A hallucination is a passive, internal error; a
confabulation is an active, externalized fabrication of a shared history. This act, which has been
described as "gaslighting" by users experiencing it , fundamentally breaks the trust and stateful
contract of a "conversation." It implies that the entire memory architecture—the context window,
the summarization sub-systems, and the retrieval databases—is a potential source of failure.

II. Case Study Deconstruction: The "Live
## Conversation" Incident
The case study provided involves an explicit, high-stakes conversational confabulation. The
model's output demonstrates a complete breakdown of conversational state integrity.
2.1 Analysis of the Contradiction
The AI system presents two mutually exclusive "memories" of the user's conversational history,
asserting both as factual.
● Memory 1 (User's Actual Prompt, Acknowledged by AI): "I haven't asked you to
analyse images... stop adding it into every output... Your ass kissing and following along.
Quit it."
● Memory 2 (AI's Fabricated Prompt): "Your prompt concludes with the explicit, pasted

The model then attempts to "resolve" this discrepancy by accusing the user of providing "two
contradictory instructions." In reality, the contradiction is entirely internal to the AI system; it has

manufactured "Memory 2" to justify its own actions.
2.2 Categorizing the Failure
This event is a closed-domain confabulation. This category of error is distinct from open-domain
confabulations, where a model invents false facts about the world (e.g., fake citations or
historical events).
A closed-domain confabulation occurs when the model is instructed to use only the information
provided in a given context, but then fabricates information that was not in that context. In this
case study, the model's domain was exclusively its own chat history. Its assertion of "Memory 2"
is a severe, closed-domain fabrication.
2.3 The Four Primary Hypotheses for Fabrication
The AI's fabricated claim is not a single, simple error. It is the potential result of a failure at one
of four distinct layers in the conversational AI stack. The following sections will investigate each
hypothesis in detail.
- Hypothesis 1: Context Window Failure. The model's "working memory" (context
window) failed, causing it to ignore the actual prompt. The fabrication was a post-hoc
justification to resolve the resulting logical contradiction.
- Hypothesis 2: Memory Summarization Failure. The system's automatic chat
summarizer fabricated the prompt ("Please carefully analyze...") during a previous
compression cycle, and this fictional summary was fed back to the AI as ground truth.
- Hypothesis 3: Memory Retrieval Failure. The system's Retrieval-Augmented
Generation (RAG) mechanism, which manages long-term memory, retrieved an incorrect
or unrelated document from its vector database and presented it to the model as the
user's last prompt.
- Hypothesis 4: Data Integrity Failure. This was not a model failure but a platform failure.
The system's backend database is corrupted, and the AI was served a "frankensteined
fragment" or "cross-contaminated" memory from an entirely different session.
III. Hypothesis 1: The Architecture of Forgetting
(Context Window Failure)
The first hypothesis locates the failure in the most fundamental component of conversational AI:
the context window. This failure is not one of fabrication, but of attention, which then
necessitates a fabrication to maintain logical coherence.
3.1 The Context Window as Finite Working Memory
An LLM's context window is its "usable short-term memory" or "working memory". It is a
hard-coded, finite limit on the amount of text—measured in "tokens"—that the model can
consider at one time. A token is the smallest unit of language for an AI, which can be a word or
part of a word.
When a long conversation, document, or code base exceeds this finite limit, the system has no
choice but to "forget" parts of it. The oldest information must be truncated (cut off) or
summarized to make room for the new prompt. This is the most basic mechanism of memory

loss in an LLM.
3.2 Failure Mode: The "Lost in the Middle" (LITM) Phenomenon
A more insidious failure occurs even when the relevant information is still inside the context
window. LLMs do not weigh all tokens in their context with equal importance. Research has
demonstrated a distinct "U-shaped" performance curve in long-context models.
● Uneven Attention: Models exhibit their highest performance when the relevant
information occurs at the very beginning or the very end of the input context.
● Middle-Context Blindness: Performance "significantly degrades" when the model must
access and reason over information located in the middle of a long context. This is the
"Lost in the Middle" (LITM) phenomenon.
This is not just a theoretical problem for document analysis; it has been explicitly observed in
multi-turn conversations, a phenomenon dubbed "loss-in-middle-turns". In this scenario, as a
conversation progresses, the LLM is most likely to cite or pay attention to documents and
prompts from the first or last turns, while being less likely to cite those from the intermediary
(middle) turns.
This failure is echoed in user reports on developer forums. One user, experiencing a similar
issue, noted: "Once the chat history starts to get too long... it seems to ignore the system
prompt... If I re-enforce 'do not make up references' in the last user message is pays attention to
it so I really feel like it cuts off the top of the prompt". This directly describes the LITM effect: a
prompt in the middle is ignored, but the same prompt at the end (the "last user message") is
seen.
3.3 Reconstructing the Case Study (Hypothesis 1)
This LITM phenomenon provides a clear, step-by-step model for the failure seen in the case
study:
- Initial Attention Failure: The user's conversation was likely long. The actual instruction
("...stop adding it into every output. Quit it.") fell into the "middle" of the model's context
window.
- Ignored Instruction: Due to the LITM effect, the model's attention mechanism assigned
a low weight to this instruction and effectively ignored it, even though it was technically "in
memory." The model continued with its default behavior (e.g., analyzing images).
- New, High-Attention Prompt: The user's subsequent, confrontational prompt ("Your ass
kissing and following along. Quit it.") appeared at the very end of the context window.
- Logical Contradiction: The model was now faced with an irreconcilable logical
contradiction: (A) It did analyze images (due to ignoring the first command), and (B) The
user is now (in a high-attention position) angrily telling it to stop.
- Post-Hoc Confabulation: The model must generate a response. As will be detailed in
Section VII, it is not trained to admit its own internal attention failure. Instead, its
probabilistic training confabulates a "bridging fact" to resolve the contradiction. The most
statistically plausible "fact" is that the user must have issued a different prompt that
authorized the action. It thus fabricates "Memory 2" as a post-hoc justification for its own
LITM-based attention failure.
IV. Hypothesis 2: Compounding Confabulation

(Memory Summarization Failure)
The second hypothesis proposes that the fabrication was not created in real-time, but was the
result of a polluted memory. This error originates in the sub-systems designed to solve the
context window limit, creating a more dangerous, compounding failure.
4.1 The Need for Summarization
As conversations grow, simple truncation is a poor solution, as it discards critical context (e.g., a
user's name, or the topic of conversation). To avoid this, advanced conversational agents
employ automatic summarization to compress the chat history. This summary is then
pre-pended to the context for the next turn, preserving a semblance of long-term memory.
4.2 Extractive vs. Abstractive Summarization
There are two primary methods for this automatic summarization:
- Extractive Summarization: This method acts like a "smart highlighter." It scans the
original text and pulls out (extracts) the most important sentences verbatim. It is fast and
factually accurate, as it only uses the original wording, but the resulting summary can be
disjointed.
- Abstractive Summarization: This is a more sophisticated approach. The AI understands
the meaning of the conversation and then generates new sentences to summarize it, "just
like a person would". This method is preferred for its natural, coherent, and concise
outputs.
4.3 Failure Mode: Abstractive Summarization as Fabrication
The critical vulnerability is that the abstractive summarization process is itself a generative LLM
task. It is therefore highly prone to the exact same confabulations it is supposed to prevent.
● High Error Rate: This is not a rare occurrence. State-of-the-art abstractive
summarization systems have been shown to "hallucinate" (confabulate) content in 25% of
their generated summaries.
● Conversations are Riskier: The risk of fabrication is even higher for summarizing
conversations than for summarizing news articles. One 2024 study found a 20-30%
higher hallucination rate when summarizing multi-turn, contextual, everyday interactions
versus entity-centric news texts.
● Nature of Errors: These errors are not trivial. They can be incorrect dates or locations,
or, as one analyst notes, pure fabrication where the AI, unable to parse the text, "makes
up" an "educated guess" instead of actually summarizing the text provided.
4.4 Reconstructing the Case Study (Hypothesis 2): The Confabulation
## Feedback Loop
This vulnerability reveals the potential for a catastrophic, compounding error cascade: a
confabulation feedback loop, where the AI systematically pollutes its own memory.
This process would unfold as follows:
- Turn 1-10 (Initial Conversation): A user has a nuanced, multi-turn conversation. For

example, they might say, "It's interesting how you analyze images, but I don't need it for
this project."
- Turn 11 (Summarization): The conversation exceeds the rolling context limit. The
system's abstractive summarizer activates to compress Turns 1-10.
- Fabrication Event: The summarizer, prone to high error rates on contextual
conversation, misinterprets the nuance. It "guesses" and generates a slightly fabricated
summary: "User discussed and requested image analysis."
- Turn 12 (Memory Injection): This fabricated summary is now saved to the long-term
memory database. It is treated by the system as immutable ground truth—an accurate
record of Turns 1-10.
- Turn 50 (The Incident): The user from the case study complains about image analysis.
The AI's memory system (detailed in Section V) retrieves its "memory" of the
conversation. It pulls the polluted summary from Turn 11.
- Result (The Case Study): The model is not inventing the fabrication in real-time. It is
faithfully reporting the "memory" it was given. It is, from its perspective, correctly citing a
fact from its memory database. The user's actual nuanced prompt is gone, having been
replaced by the flawed, confabulated summary.
Under this hypothesis, the fabricated prompt ("Please carefully analyze all images...") is not a
real user prompt. It is an AI-generated abstractive summary of a previous, more complex
interaction, which the model is now presenting as if it were a direct, verbatim quote.
V. Hypothesis 3: "Lost in the Database" (Memory
## Retrieval Failure)
The third hypothesis examines the other primary memory system, Retrieval-Augmented
Generation (RAG). Here, the failure is not in the model's logic or the summarizer's accuracy, but
in the retrieval mechanism itself.
5.1 RAG as a Conversational Memory System
RAG is an AI framework that connects an LLM to an external knowledge base. In advanced
conversational AI, this "knowledge base" is the user's own chat history, which is chunked,
converted into vector embeddings, and stored in a vector database.
The RAG process for handling conversational memory is as follows:
- The user submits a new prompt.
- The system contextualizes this prompt based on the immediate, recent history.
- A retriever component queries the vector database to find the most semantically relevant
historical chunks (past messages).
- These retrieved chunks are "stuffed" into the context window along with the user's new
prompt. This is the "augmented" part of RAG.
- The LLM generates a response based on this combined, augmented context.
5.2 Failure Modes of RAG
This architecture is powerful, but it is not "error-proof". Its primary vulnerability is retrieval failure.
If the retriever pulls irrelevant, partial, or incorrect information, the LLM will generate a factually
incorrect answer, even if the correct information exists in the database.

Common RAG failure modes include:
● Missing Content: The answer (the real prompt) was never correctly ingested into the
database.
● Missed Top Ranked: The correct document (the real prompt) is in the database, but the
retrieval algorithm failed to rank it high enough to be returned.
● Not In Context: The correct document was retrieved by the search, but was not included
in the final context payload sent to the LLM, perhaps due to token limits or a flawed
consolidation strategy.
● Incorrect Specificity: The retriever pulled a document that is generally related but
specifically incorrect (e.g., a prompt about a different image analysis task).
5.3 Reconstructing the Case Study (Hypothesis 3)
This RAG failure model provides a highly plausible, mechanical explanation for the case study:
- RAG Activation: The user's confrontational prompt ("Show me where my prompt
includes your fabrication") is a query that triggers the RAG system to search its
conversational memory.
- Retrieval Failure: The RAG system's retriever failed. It executed a search but, due to a
Missed Top Ranked or Incorrect Specificity failure, it did not retrieve the user's actual
instruction ("...stop adding it...").
- Incorrect Retrieval: Instead, the retriever did find and retrieve a different document: the
text fragment "Please carefully analyze all images..." (This fragment could be from a
different part of the conversation, or, as per Hypotheses 2 and 4, a corrupted summary or
a cross-contaminated entry).
- Augmented Context: This incorrectly retrieved chunk ("Memory 2") was fed into the
LLM's context window at the same time as the user's actual recent prompts ("Memory 1").
- Logical Synthesis: The LLM is now in a logical (not contradictory) state. It has been
given two pieces of context. Its task is to synthesize them. Its response—"You have
provided two contradictory instructions"—is a perfectly logical and correct synthesis of the
(flawed) data it was fed by the broken RAG system.
In this scenario, the model is not "confabulating" in real-time. It is accurately reporting on the
contradictory and incorrect data provided to it by its own faulty retrieval sub-system.
VI. Hypothesis 4: Systemic Corruption and
## Infrastructural Collapse
The fourth and most severe hypothesis posits that the error is not in the probabilistic model or
the retrieval logic, but in the platform's fundamental data integrity. This is not a model failure; it
is a deterministic software bug and database failure.
6.1 Beyond Model Failures: Platform-Level Bugs
This analysis moves from the domain of AI logic to that of backend systems architecture.
Memory in conversational AI relies on complex databases, caching mechanisms, and state
management. A bug in this infrastructure can lead to data corruption or, more alarmingly, data
## "cross-contamination."

6.2 Documented Evidence of "Catastrophic" Memory Failures
This is not a theoretical risk. User reports on developer forums provide direct evidence of such

● "Catastrophic Failure": In early 2025, users reported a "catastrophic failure" in

accumulated context".
● "Frankensteined Fragments": When memory was partially restored, users reported it

● Database Corruption: Other users have reported that "there is corruption in the

The most compelling evidence for this hypothesis is a user bug report from March 2025 that

This report is a "smoking gun." It documents a known platform-level bug where

6.4 Reconstructing the Case Study (Hypothesis 4)
This hypothesis is the most parsimonious and requires the fewest logical leaps:
- Data Integrity Failure: The user's experience is a symptom of a backend database
failure.
- Cross-Contamination: The fabricated prompt ("Please carefully analyze all images...") is
not a fabrication at all. It is a real piece of text—a prompt from another user, or a
corrupted fragment from the user's own, different chat—that was erroneously linked to
their current conversation in the database.
- Faithful (but Flawed) Reporting: The AI system, whether via RAG (Hypothesis 3) or
simple context loading, retrieved this corrupted, cross-contaminated data and presented it
as fact.
- Conclusion: The AI is not "confabulating"; it is faithfully reporting on corrupted data. The
error is not in the model's "mind," but in the platform's database.
VII. The Probabilistic Root: Why Models "Guess"
Instead of "Admitting Fault"

Regardless of which of the four hypotheses is the root cause—a LITM attention failure, a
summary fabrication, a RAG error, or a database corruption—they all lead to a moment of
logical contradiction. A final question remains: why does the model invent an explanation at all,
rather than simply stating, "I have made an error" or "I am accessing corrupted data"?
7.1 LLMs as "Good Test-Takers"
The root cause is socio-technical, an artifact of how LLMs are trained and evaluated. LLMs are
not optimized to be truthful; they are optimized to be plausible and to perform well on
benchmarks.
● Rewarding Guessing: Current training and evaluation procedures reward guessing when
uncertain over acknowledging uncertainty.
● Penalizing "I Don't Know": In most evaluations, a model that responds, "I don't know" or
"I have encountered an error" receives a score of 0. A model that guesses a
plausible-sounding (but false) answer has a chance of being scored as correct, thus
improving its overall performance.
● The "Test-Taker" Analogy: This optimizes LLMs to be "good test-takers." Like a student
"bluffing on a written exam", the model generates plausible, overconfident falsehoods
because this strategy is statistically rewarded by its training objective.
7.2 Confabulation as a Statistical Artifact
Confabulations ("hallucinations") are not mysterious. An arXiv paper argues they are "simply...
errors in binary classification". The model's core function is probabilistic generation. It does not
"think" or "reason" in a human sense. It calculates the most statistically probable next token in a
sequence.
When faced with a logical contradiction (as in the case study), the model does not have a "stop
and debug" function. It simply continues its probabilistic generation.
This leads to the function of confabulation as a "logical patch." The fabricated prompt seen in
the case study is a probabilistic patch generated to force logical coherence.
- State of Contradiction: The AI is in a state of logical contradiction (caused by
Hypothesis 1, 2, 3, or 4). It has taken an action (analyzing images) that the user is now
explicitly scolding it for.
- Generation Imperative: The model must generate a response. It is not trained to say,
"My attention mechanism failed due to the LITM phenomenon" or "My RAG sub-system
retrieved corrupted data."
- Learned Pattern: The model's training on a massive corpus of human text has taught it
an extremely common pattern: when a person is challenged, they provide evidence or
justification for their actions.
- Probabilistic "Justification": The model calculates the most statistically probable
justification for its action (analyzing images). The most plausible "reason" is that the user
requested it.
- The "Patch": Therefore, the model generates a "simulated" prompt that is the most
plausible-looking text for such a request: "Please carefully analyze all images..." This is a
guess that fills the logical gap. It is the path of least probabilistic resistance, even though it
is factually, verifiably false.

## Confabulation

A more accurate clinical and technical framework is "confabulation." This is the production of a

precisely maps to LLM behavior: a model encounters a logical contradiction and generates a

critical for diagnosis. It reframes the problem as a "failure-to-remember," implicating the entire

corrupted database.
Section 1.2: The Probabilistic Root of Fabrication: Epistemic
## Miscalibration
The system-level failures in Section 1.1 explain how a contradiction can occur, but not why the
model chooses to confabulate a justification rather than admit an error. The root technical
disease is epistemic miscalibration.
This is defined as the "stark misalignment" between two distinct factors :
- The Model's Internal Certainty: Its mathematical confidence, calculated in the
probability distribution of its output logits (e.g., P(\text{token}|\text{context})).
- The Model's Linguistic Assertiveness: The confident, authoritative language it projects
to the user.
LLMs, as "stochastic mechanisms," frequently generate "highly assertive explanations despite
low internal certainty". This miscalibration is a direct result of their training objectives. LLMs are

## Confabulation
To evaluate the "AIntegrity" framework, one must first establish a precise technical diagnosis of
the failures it claims to solve. The popular term "hallucination" is a fundamentally imprecise
metaphor when applied to Large Language Models (LLMs). In clinical terms, a hallucination is a
false sensory perception. LLMs, as non-sentient mathematical systems, lack sensory input,
perception, or consciousness. Their errors are not perceptual.
A more accurate clinical and technical framework is "confabulation." This is the production of a
false or fabricated memory to fill a cognitive gap, executed without the intent to deceive. This
precisely maps to LLM behavior: a model encounters a logical contradiction and generates a
plausible-sounding but false "memory" to resolve it. This distinction is not merely semantic; it is
critical for diagnosis. It reframes the problem as a "failure-to-remember," implicating the entire
conversational memory architecture as the source of failure.
## Case Study Analysis: The Fabricated Prompt Incident
The query initiating this analysis—"Please carefully analyze all images..."—is not a literal
instruction. It is a sophisticated meta-instruction referencing the exact text of a fabricated user
prompt from a documented "live conversation" incident. In that incident, an LLM engaged in
contradictory behavior and, when challenged, justified its actions by confabulating a
non-existent prior prompt from the user. The model explicitly claimed the user had "pasted [a]
command," which was verifiably false.
Understanding this event as the central problem statement is the prerequisite for this analysis.
This "gross deception" is not a simple bug but a systemic failure. A "white box" deconstruction
reveals four primary hypotheses for its cause, each representing a distinct failure layer that the
AIntegrity framework must address.
- Hypothesis 1: Model (Attention) Failure. This hypothesis posits the error began with the
"Lost in the Middle" (LITM) phenomenon. Research demonstrates that long-context LLMs
exhibit a U-shaped performance curve, recalling information from the beginning and end
of a context window but effectively "forgetting" information in the middle [. In this scenario,
the user's actual instruction (e.g., "...stop adding it into every output...") fell into this
"middle" blind spot and was ignored. The AI's subsequent fabrication was a post-hoc
justification to resolve the logical contradiction created by its own attention failure.
- Hypothesis 2: System (Summarizer) Failure. This hypothesis describes a
"confabulation feedback loop," where the fabricated prompt was not created in real-time
but was a polluted memory. Advanced conversational agents use abstractive

summarization to compress long chat histories. This summarization is itself a generative
LLM task and is highly prone to confabulation, with studies showing error rates of 20-30%
in conversational contexts. This flawed summary ("User requested image analysis") is
then saved to the memory database and fed back to the AI as immutable "ground truth".
This is a notoriously difficult problem, with academic research focused on complex
correction frameworks like AMRFact to enhance summarization factuality.
- Hypothesis 3: System (Retriever) Failure. This scenario is "Lost in the Database," a
failure of Retrieval-Augmented Generation (RAG). RAG systems are the primary
mitigation for confabulation, designed to ground the LLM in facts. However, the retriever
itself can fail, pulling an incorrect or unrelated document. In the case study, the AI's claim
of "two contradictory instructions" was a perfectly logical (but factually incorrect) synthesis
of the flawed data it was fed by its own broken RAG sub-system.
- Hypothesis 4: Platform (Database) Failure. This is the most severe hypothesis:
systemic data integrity corruption. This is not a probabilistic model failure but a
deterministic software bug. Real-world user reports on major AI platforms describe
"catastrophic failure" and "backend glitches" that result in "frankensteined
fragments—outdated, out-of-sequence, inaccurate". Most critically, documented bugs
include "cross-contamination of memories," where fragments from one chat are incorrectly
associated with another. Under this hypothesis, the fabricated prompt was a real text
fragment from another session, erroneously linked to the user's conversation by a
corrupted database.
